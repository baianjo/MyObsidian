### 📊 总结：属于你的指挥面板

最后，你可以把这个表格保存在手边，作为你日常开发的**战术对照表**：

| **战术技能**                  | **动作类比** | **短期/小工具项目** | **长期/大型底座项目**   |
| ------------------------- | -------- | ------------ | --------------- |
| **`grill-with-docs`**     | 盘问排雷     | 第一天用一次       | 每次大版本更新前用       |
| **`to-prd`**              | 画蓝图      | 第一天用一次       | 每次立项核心文档        |
| **`to-issues`**           | 切砖块      | 切出少量大块       | 切出大量垂直切片        |
| **`zoom-out`**            | 看地图      | 偶尔防跨域/通信死锁   | **极高频**，防改坏底层   |
| **`tdd`**                 | 稳扎稳打     | 仅用于复杂数据解析    | **核心标配**，全覆盖重灾区 |
| **`diagnose`**            | 科学修车     | 修玄学环境 Bug    | 修复杂竞态/状态流转 Bug  |
| **`improve-codebase...`** | 大扫除      | ❌ 基本不用       | ✅ 每月/季度定期大保养    |
| **`handoff`**             | 存档交接     | 简单备忘         | **核心记忆库**，防失忆断层 |




### 🟢 阶段一：立项与施工图阶段 (项目第 0 天)

_这个阶段你不需要写哪怕一行代码。这里的目标是“防瞎想、定规矩”。_

#### 1. `grill-with-docs` (架构师的灵魂拷问)

- **什么时候用：** 当你脑子里只有一个Idea，或者刚写完一个粗略的 README，准备正式开干的第一句话。
    
- **怎么用：** 把你的初步想法丢给 AI，让它盘问你。它会逼着你思考边界条件、技术栈限制和外部依赖，最后生成 `CONTEXT.md` 和架构决策记录 (ADR)。
    
- **周期差异：**
    
    - **短期项目（如插件）：** 重点盘问环境限制（如 Manifest V3 的权限怎么申请、跨域怎么解决）。
        
    - **长期项目（如 MaaPlus）：** 重点盘问数据结构的扩展性、状态机的解耦、以及接口未来如何对接 AI。
        

#### 2. `to-prd` (产品经理写蓝图)

- **什么时候用：** `grill-with-docs` 问答结束，大家达成共识之后。
    
- **怎么用：** 指令 AI：“把刚才聊的共识，固化成一份产品需求文档 (PRD)”。这相当于给本次开发立下“军令状”，明确什么做、什么绝对不做（Non-goals）。
    
- **周期差异：**
    
    - **短期项目：** 产出一份一页纸的文档，明确 Happy Path（主流程）和 Fallback（降级方案）即可。
        
    - **长期项目：** 产出极其严谨的文档，需要精确定义脱敏级别、异常接管的重试次数等硬性指标。
        

#### 3. `to-issues` (项目经理派发砖块)

- **什么时候用：** PRD 生成完毕，你准备开始真正动手写代码之前。
    
- **怎么用：** 指令 AI：“根据 PRD，拆解出我可以独立执行的垂直切片任务”。这会把大饼切成可以一口吃掉的工单（Issue）。
    
- **周期差异：**
    
    - **短期项目：** 拆解为 3-5 个简单的组件任务（UI、后台服务、解析器）。
        
    - **长期项目：** 可能会拆解出长达一个月的任务清单，你需要按优先级（如先搭本地调度，再做 UI）一个个推进。

### 🟡 阶段二：实质性编码与攻坚 (日常搬砖阶段)

_这个阶段是你每天花时间最多的地方，核心目标是“防改坏、防 Bug”。_

#### 4. `zoom-out` (高空看地图)

- **什么时候用：** 当你要在一个**已经有一定代码量的项目**里，修改一个核心逻辑，或者当你不知道该把新文件建在哪个目录下时。
    
- **怎么用：** 动手前先执行 `zoom-out`。让 AI 扫描当前目录结构，画出模块调用关系。
    
- **周期差异：**
    
    - **短期项目：** 偶尔用来检查一下消息传递（Message Passing）的链路有没有死锁。
        
    - **长期项目：** **极其高频使用。** 每次接手新模块前必须看地图，防止 AI 给你瞎提建议，把底层执行器和上层调度器搞耦合。
        

#### 5. `tdd` (测试驱动推进器)

- **什么时候用：** 当你开始做一个具体的 Issue，且这个 Issue 包含**纯数据处理、核心业务逻辑、或者极易出错的状态流转**时。
    
- **怎么用：** 让 AI 先写测试用例（红），再写业务代码（绿），最后优化（重构）。
    
- **周期差异：**
    
    - **短期项目：** 只在处理复杂外部数据时用（比如 B 站经常变动的字幕 JSON 解析）。简单的 UI 渲染不需要 TDD。
        
    - **长期项目：** **核心组件的标配。** 账号管理器、模拟器状态监测等底座代码，必须用 TDD 夯实，否则后期牵一发而动全身。

### 🔴 阶段三：战损修复与架构保养 (踩坑与维护阶段)

_代码跑不通了，或者代码写得太恶心了，这时候必须停下来。_

#### 6. `diagnose` (科学修车仪)

- **什么时候用：** 当程序报错、白屏、死循环，或者出现了你肉眼看不出原因的玄学 Bug 时。（**严禁直接把报错贴给 AI 喊它 "Fix it"**）。
    
- **怎么用：** 告诉 AI 症状，执行 `diagnose`。强迫它走：列出假设 -> 提出日志插桩方案 -> 验证日志 -> 最终修复 的科学流程。
    
- **周期差异：**
    
    - **短期项目：** 解决平台特有 Bug（比如 Chrome 插件 Service Worker 的幽灵休眠）。
        
    - **长期项目：** 解决复杂的并发竞态条件（比如两个账号同时请求模拟器句柄）。
        

#### 7. `improve-codebase-architecture` (架构大扫除)

- **什么时候用：** 当项目开发了几个月，你发现加一个新功能越来越吃力，或者各种 `utils.js` 塞满了数百个杂乱的函数时。
    
- **怎么用：** 这是一个**专题活动**。周末倒杯咖啡，执行这个技能，让 AI 帮你找出“深层模块”重构机会，把腐化的代码重新梳理干净。
    
- **周期差异：**
    
    - **短期项目：** 基本上 **不需要使用**。代码跑通就完事了，没必要过度重构。
        
    - **长期项目：** **项目续命的关键。** 每隔 1-2 个月必须做一次，为下一阶段（比如 MaaPlus 接入真正的 AI 大模型）腾出干净的接口空间。

### ⏸️ 阶段四：跨周期与防失忆 (进度存档)

_你的大脑和 AI 的上下文窗口都是有极限的。_

#### 8. `handoff` (意识存档提取)

- **什么时候用：** 今天干累了要睡觉、周末要出去玩、或者当前会话的上下文太长导致 AI 开始胡言乱语需要开新窗口时。
    
- **怎么用：** 睡前执行 `handoff`，让 AI 写下一份翔实的当前状态文档（做到了哪、卡在了哪、明天第一步干啥）。明天开新窗口，把这个文档扔给新会话。
    
- **周期差异：**
    
    - **短期项目：** 简单记录“今天完成了前端，明天搞 API”即可。
        
    - **长期项目：** **无价之宝。** 它可以记录下你试错失败的教训（比如“千万别用方法 A 杀进程，会留僵尸进程”），让新会话的 AI 绝对不踩昨天的坑。
