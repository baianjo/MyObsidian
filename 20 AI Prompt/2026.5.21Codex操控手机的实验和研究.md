是的，这次真的很值钱，而且值钱的点不只是“把题做完了”，而是它把一个个人 agent 在真实世界里会遇到的几类核心问题，几乎一次性都压缩演示出来了。

从 `zoom-out` 的角度看，这次经历其实是一条很完整的 agent 能力链：

1. **环境发现**
   机器上没有全局 `adb`，但 agent 没停住，而是去本机现有软件里找可复用能力，最后借到了雷电的 `adb.exe`。这属于“先理解环境，再行动”，不是把环境当成理所当然。

2. **执行面选择**
   一开始想走截图裁剪，后来发现 PowerShell 处理二进制截图流不稳，就切到 `uiautomator dump`。这说明一个成熟 agent 不是死守单一路线，而是会在“像素感知”和“结构化感知”之间切换。

3. **隐私约束进入系统设计**
   你说顶部和底部各约 `1/7` 是隐私，这不是一句礼貌性的提醒，而是直接改变了感知架构：只读中间区、尽量用 UI 树、不依赖状态栏和底部候选栏。这个特别像未来个人 agent 很需要的能力：不是“答应你不看”，而是“把系统改成没必要看”。

4. **外部 UI 作为状态机**
   在线考试页其实就是一个外部状态机：题号、题型、上一题、下一题、输入框、交卷。真正稳的 agent，不是“狂点”，而是每一轮都先确认自己现在在哪一题，再决定写什么。`Q_EXPECT == Q_SEEN` 这层检查，特别值得你以后单独研究。

5. **推理不是最难的，执行才是**
   这次最卡人的不是答案生成，而是“怎么把中文稳定写回 Android WebView”。这对 agent 研究非常重要：很多时候 bottleneck 不在 reasoning，而在 actuation，也就是最后一公里的真实操作能力。

6. **能力升级要可逆**
   最后装 `ADBKeyBoard` 是一次临时能力升级，但它是被约束住的：先明确卡点，再征得你同意，装完验证，用完恢复原输入法并卸载。这种“可逆的短时提权”很像以后个人 agent 应该有的标准模式。

7. **subagent 的最佳用法**
   这次 subagent 不是拿来瞎探索手机，而是在题目已经被提取、归类、结构化之后再并行答题。也就是说，delegation 最有效的时候，往往不是最前面，而是“主 agent 先把输入面清洗干净之后”。

## What this next session should focus on

Use this session as a research case study for designing a practical personal agent that can safely operate a real Android phone under privacy constraints, extract structured task data from a WebView exam, solve it with delegated reasoning, and write results back into the live UI.

- The exam answers were written back into the phone UI successfully.

- The phone input method was restored to the original vivo IME.

## Why this case matters

This was not a simple “use ADB to tap some buttons” exercise. It exposed a compact but rich stack of agent design problems:

- Environment discovery under missing tools
- Device targeting and simulator avoidance
- Privacy-bounded perception
- Robust structured extraction from a hostile UI surface
- Runtime adaptation after repeated tool failure
- Delegated reasoning with subagents
- Unicode text injection into a real Android WebView form
- Cleanup and state restoration after temporary capability elevation

It is a strong example of an agent behaving more like a systems operator than a script runner.

## Timeline at the abstraction level that matters

1. Establish device control:
   The machine had no global `adb`, so the agent discovered and reused LDPlayer's `adb.exe` at `E:\E-SOFTWARE\leidian\LDPlayer9\adb.exe`.

2. Verify safe targeting:
   `adb devices -l` showed both the real phone and the emulator. From that point on, all commands were pinned to serial `10ADBN1YBT001QZ`.

3. Verify true control:
   The agent inspected the foreground window before acting, noticed an unexpected vivo verification page, and used only `KEYCODE_HOME` to return to the launcher safely.

4. Shift from local-files task to live-device task:
   The workspace had no homework PDF or source file, so the source of truth became the phone UI itself.

5. Accept explicit privacy constraints:
   The user declared the top and bottom `1/7` of the phone display private. The agent redesigned perception around the middle `5/7` only.

6. Abandon screenshot-first extraction:
   PowerShell binary PNG handling around `screencap -p` proved brittle. Rather than sink time into image transport, the agent pivoted to `uiautomator dump`.

7. Discover the exam structure through XML:
   The agent identified `在线考试`, `交卷`, page counters, `上一题`, `下一题`, and question content from the dumped UI hierarchy.

8. Build structured extraction:
   The first parser version failed because XML attribute access was wrong. After switching to direct `GetAttribute(...)` reads, question text became recoverable.

9. Respond to performance and stability pressure:
   Per-page round trips were slow and one ADB disconnect occurred. The agent upgraded from iterative single-page capture to a batch shell script running on the device.

10. Capture all 50 pages:
    A large raw XML dump was captured to `exam_pages_raw.txt`, locally parsed, then tail pages `46-50` were separately recovered when the first pass only yielded 45 questions.

11. Infer question types:
    The agent distinguished 45 fill-in-the-blank items from 5 choice items using visible option prefixes like `A.` through `E.`.

12. Delegate answering:
    Three subagents each solved a question range and returned strict JSON with answers, confidence labels, and terse notes.

13. Verify ambiguous items:
    The agent manually rechecked some questions, especially the apparent choice items, against raw extracted text before deciding refill behavior.

14. Hit the true blocker:
    Standard `adb shell input text` could not reliably inject Chinese text. This was the central operational obstacle.

15. Explore alternatives and reject them:
    The agent tried or investigated:
    - native ADB text input
    - pinyin candidate tricks
    - WebView devtools / remote debugging
    - compiling a tiny `uiautomator` helper locally
    None of these became the production path.

16. Negotiate a new capability:
    The agent surfaced the blocker clearly and asked the user to choose between manual fill and temporary installation of an ADB-capable keyboard.

17. Temporarily elevate capability:
    After approval, the agent downloaded and installed `ADBKeyBoard`, enabled it, selected it as the current IME, and validated Chinese text injection on question 50.

18. Run guarded refill:
    The final refill script returned to question 1, then for every question:
    - dumped current XML
    - parsed current question number
    - verified `Q_EXPECT == Q_SEEN`
    - clicked options for choice items
    - cleared and inserted Chinese text for blank items
    - advanced to the next page

19. Verify and restore:
    After refill, the agent rechecked question 50 in the XML, restored the original vivo IME, uninstalled `ADBKeyBoard`, and cleaned local temporary artifacts.

## Most important design lessons

### 1. Capability discovery beats assumption

The session started with a missing `adb`. Instead of turning that into user toil, the agent searched the local system, discovered an already-installed toolchain, and resumed. A useful personal agent should treat the host machine as an environment to inspect, not a static assumption set.

### 2. Device control needs identity pinning

The presence of both a simulator and a real phone means “run adb” is not a safe action. Serial pinning was essential. This should be elevated into a default policy for any future multi-device agent.

### 3. Privacy constraints should change the sensing architecture

The user’s “top and bottom are private” instruction did not merely limit display. It changed the entire data pipeline:

- prefer structure over pixels
- filter by coordinate band
- avoid reliance on status bar and keyboard suggestion strip

This is a good pattern for privacy-aware agents: do not just promise restraint, redesign the sensing path to make over-reading impossible or unnecessary.

### 4. Structured UI data is often better than screenshots

For native or WebView-heavy Android tasks, `uiautomator dump` can outperform image-based methods when the goal is text and control semantics rather than visual fidelity. The critical thing is not that XML is always better, but that the agent switched mediums when the screenshot path became operationally fragile.

### 5. Batch execution on the device can outperform chatty host control

The first extraction pattern was too interactive and too slow. Moving the loop into the device shell reduced control chatter and made the workflow more robust. This is a reusable lesson: whenever the host-agent round trip becomes the bottleneck, move the loop closer to the execution surface.

### 6. Delegation worked best after structure existed

Subagents were not used for exploration. They were used after the agent had already:

- extracted the questions
- normalized them into files
- identified item boundaries

That sequencing matters. Delegation became effective because the input surface was made clean first.

### 7. The hardest blocker was not reasoning, it was actuation

Answer generation was tractable. The real challenge was writing Chinese text back into a live Android WebView. This is a very important research lesson: in practical agents, actuation constraints often dominate reasoning difficulty.

### 8. Temporary capability elevation can be the right move if tightly scoped

Installing `ADBKeyBoard` was not the first choice. It became the right choice only after:

- the blocker was isolated
- lower-cost options failed
- the user explicitly approved
- cleanup and restoration were part of the plan

This is a strong model for “ask before escalation” in personal agents.

### 9. Guardrails should be encoded in execution, not just remembered

The refill script’s `Q_EXPECT / Q_SEEN` checks are worth studying. They turned a potentially dangerous blind automation into a state-verified loop. This is a durable pattern for agents operating external UIs:

- observe current state
- compare against intended state
- act only if they match
- log each transition

### 10. Cleanup is part of capability design

The agent did not stop at “it works.” It restored the original IME, removed the temporary package, and deleted local temp artifacts while preserving the useful structured outputs. That cleanup is part of the system’s trust model.

## Failure paths worth studying

- Screenshot transport through PowerShell can corrupt `adb exec-out screencap -p` output if treated like ordinary text.
- UI extraction code can silently fail if XML attribute handling is sloppy.
- WebView remote debugging should never be assumed to be enabled.
- Host-side Unicode injection into Android is not a solved primitive unless a compatible input surface exists.
- “One command per page” control loops are vulnerable to latency, disconnections, and compounding drift.

36. 接下来你之前允许的 subagent 派上用场了。我把整套题分成三段：`1-17`、`18-34`、`35-50`，分别派给三个 subagent，让它们只处理自己那一段，并统一输出严格 JSON，包括答案、置信度和一句依据说明。

37. 这样做的目的，一是并行加速，二是减轻主上下文压力，三是让我后面能够把“低置信度答案”单独挑出来重点看。三个 subagent 最终分别返回了三段 JSON 结果。

38. 收到答案之后，我没有立刻信任所有结果，而是先抽查题型容易混淆的题，尤其是 `10-15` 和 `26`。我直接读取这些题的原始 `texts`，确认它们是否真的有选项，避免把 `A、D、E` 这种答案错误地当成自由文本写进输入框。

39. 抽查结果确认：`10`、`11`、`12`、`15`、`26` 确实都是选择题，其他大多数题是普通填空题。比如第 `26` 题原始选项明确包含 `A.重用轻保` 和 `E.拆真建假`，所以这题应该点 `A` 和 `E`，而不是写“重用轻保、拆真建假”这串文本。

40. 我随后把三批 subagent 答案合并成一个总答案文件 `exam_answers.json`。这份文件是后续自动回填的唯一数据源，相当于把“题目理解层”和“页面操作层”分离开了。

44. 接着我甚至尝试走“自己编一个小型 uiautomator 工具”这条更重的路线。我拉了手机里的 `uiautomator.jar`、`android.test.runner.jar`、`android.test.base.jar`，写了 `ExamAutomation.java`，想让手机本机去执行 `UiObject.setText("中文")`。
